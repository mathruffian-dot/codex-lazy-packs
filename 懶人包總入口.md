# Codex 懶人包總入口

> 用這份決定「先跑哪一包、後跑哪一包」。單篇懶人包負責細節，這份只負責整合路線。

---

## 先講結論

多數老師或教學工具專案，建議照這條主線：

1. `00-環境建置.md`
2. `01.5-Codex必裝Skills與Plugins.md`
3. `02-連接-GitHub.md`
4. `03-建立第二大腦-Obsidian.md`
5. `04.5-連接-Firebase-資料庫.md`
6. `07-初始化班級工具工作模式.md`
7. 之後依需求加 `08-用Image Gen Skill在Codex生圖.md`、`09-用chezmoi同步Codex設定.md`

這樣會得到：

- Codex 可用
- GitHub 可 commit / push / Pages
- Obsidian 可當專案駕駛艙
- Firebase 可做即時互動資料庫
- 專案有 `AGENTS.md`、README、`.gitignore`、開工 / 收工流程

---

## 不要一次全跑

這樣做的風險是：工具太多、權限太多、設定互相影響，出錯時不知道是哪一步壞掉。

建議改成：

- 先跑「必要主線」
- 每跑完一包就驗證
- 等第一個教學工具真的做出來後，再補進階包

---

## 路線 A：第一次使用 Codex

適合：新電腦、新帳號、第一次安裝。

| 順序 | 懶人包 | 完成標準 |
|---|---|---|
| 1 | `00-環境建置.md` | Git、GitHub CLI、Node / npm 可用 |
| 2 | `01.5-Codex必裝Skills與Plugins.md` | 基礎 plugins / skills 可用 |
| 3 | `02-連接-GitHub.md` | `gh auth status` 正常，能 push |
| 4 | `03-建立第二大腦-Obsidian.md` | Codex 找得到 vault |
| 5 | `09-用chezmoi同步Codex設定.md` | 多電腦才需要；單機可先跳過 |

---

## 路線 B：建立班級互動工具專案

適合：課堂投票、即時文字雲、遊戲計分、表單展示、學生互動頁。

| 順序 | 懶人包 | 完成標準 |
|---|---|---|
| 1 | `02-連接-GitHub.md` | repo 可建立、可 push |
| 2 | `04.5-連接-Firebase-資料庫.md` | Firestore 可新增 / 讀取 / 刪除測試資料 |
| 3 | `07-初始化班級工具工作模式.md` | 專案有 `AGENTS.md`、README、Obsidian 駕駛艙、GitHub repo |
| 4 | 影片實作段落 | 做出第一個工具 |

建議預設：

- 資料庫：Firebase
- repo：如果要 GitHub Pages，通常用公開 repo
- 學生資料：只存座號與班級代號，不存真名

---

## 路線 C：第二大腦與專案知識管理

適合：想讓 Codex 記住專案進度、踩坑、下一步。

| 順序 | 懶人包 | 用途 |
|---|---|---|
| 1 | `03-建立第二大腦-Obsidian.md` | 建 vault 與基本規則 |
| 2 | `04-第二大腦設定指南.md` | 設計三層結構與模板 |
| 3 | `02.5-連接GitHub與Obsidian.md` | 把 GitHub 工作紀錄接到 Obsidian |
| 4 | `07-初始化班級工具工作模式.md` | 每個專案建立自己的駕駛艙 |

分工原則：

- `AGENTS.md`：固定規則、路徑、專案邊界
- Obsidian 駕駛艙：進度、下一步、踩坑、收工紀錄
- GitHub：版本與部署

## 路線 D：替網頁工具加 AI 能力

適合：已經有網頁工具，想加本地 AI 或免費 API。

| 選項 | 懶人包 | 適合情境 |
|---|---|---|
| 本地模型 | `05-安裝本地AI-Ollama.md` | 不想把資料送雲端、可接受本機資源消耗 |
| 免費雲端 API | `06-設定Gemini免費API.md` | 想快速接 API、可接受外部服務限制 |
| 生圖 | `08-用Image Gen Skill在Codex生圖.md` | 製作課堂素材、插圖、封面、活動圖 |

注意：`05`、`06` 是給「你做出來的工具」用，不是 Codex 本身必裝。

---

## 資料庫怎麼選

| 情境 | 建議 | 原因 |
|---|---|---|
| 課堂即時互動、多人同時輸入 | Firebase | 即時更新友善，不會閒置暫停 |
| SQL 統計、關聯查詢、資料分析 | Supabase | PostgreSQL / SQL 比較適合 |
| 只做靜態網頁 | 先不要資料庫 | 少一層故障點 |

不要同時一開始就接 Firebase 和 Supabase。先選一個，等需求明確再換。

---

## GitHub Pages 怎麼選

| 需求 | 建議 |
|---|---|
| 要公開展示給學生 | 公開 repo + GitHub Pages |
| 不想公開程式碼 | 私有 repo；改用其他部署方式，或確認 GitHub 方案是否支援私有 Pages |
| 只是備份版本 | 私有 repo，不需要 Pages |

如果 GitHub 回：

```text
Your current plan does not support GitHub Pages for this repository.
```

代表不是設定錯，而是「私有 repo + Pages」受方案限制。

## 已存在專案怎麼套

不要重跑完整初始化。

先檢查：

- 是否已有 Git repo
- 是否已有 GitHub remote
- 是否已有 `AGENTS.md`
- 是否已有 README
- 是否已有 `.gitignore`
- 是否已有 Firebase / Supabase 設定
- 是否已有 Obsidian 駕駛艙

只補缺口。不要覆蓋既有網頁、資料庫規則、Git 歷史或已寫好的專案規則。

---

## 新專案完成標準

完成 #07 後，專案至少要有：

- `AGENTS.md`
- `README.md`
- `.gitignore`
- Git repo
- GitHub repo
- Obsidian 駕駛艙
- 若用資料庫：`.firebaserc` / `firebase.json` / `firestore.rules` 或 Supabase 對應設定
- 可說「開工」讀進度
- 可說「收工」更新進度

## 常見錯誤決策

| 錯誤做法 | 風險 | 建議 |
|---|---|---|
| 一開始把所有包都跑完 | 出錯難排查 | 先跑主線，後補進階 |
| 把 Obsidian 駕駛艙建在 repo 裡 | 進度與程式碼混在一起 | 建在 Obsidian vault |
| `AGENTS.md` 寫每日進度 | 固定規則會被污染 | 進度寫 Obsidian |
| 要 Pages 又堅持 private repo | 可能受 GitHub 方案限制 | 公開 repo 或換部署方式 |
| demo Firestore rules 長期公開 | 任何人可寫資料 | 正式服務改登入 / 班級碼 / 教師管理 |
| 專案用途只寫一兩個字 | Codex 不知道邊界 | 補成一句可判斷範圍的描述 |
