## 這個懶人包會幫你做什麼？

這是一份「工作模式啟動包」。跑完之後，你就可以用 Codex 管理一個教學應用專案：

- 建好 Google Drive / OneDrive 裡的工作資料夾
- 建好 `AGENTS.md`，讓 Codex 每次都知道專案規則
- 建好 Obsidian 專案駕駛艙，記錄進度、下一步、踩坑
- 建好 GitHub repo，必要時啟用 GitHub Pages
- 套用 Codex 的三個全域流程：
  - `project-init-sync`：新專案初始化
  - `startup-sync`：開工接續
  - `shutdown-sync`：收工同步
- 讓專案準備好，接著回到影片教學，跟著做第一個工具

> 一句話：Claude 版靠 `/開工`、`/收工` skills；Codex 版改用內建的 `開工`、`收工`、`新專案初始化` 三個全域 skill 流程。

## GitHub repo 建立流程

若要用 GitHub Pages，建議公開 repo。

PowerShell 範例：

```powershell
Set-Location "G:\我的雲端硬碟\<專案名稱>"
git init
git config windows.appendAtomically false
git add AGENTS.md README.md .gitignore
git commit -m "初始化教學工具專案"
git branch -M main
gh repo create <repo名> --public --source=. --push
```

如果不需要 Pages，可以改成私有：

```powershell
gh repo create <repo名> --private --source=. --push
```

> Google Drive 內的 git repo 建議先設 `git config windows.appendAtomically false`，可避免 `cannot update the ref` 之類的同步衝突。

## 2026-06-10 實測修正：從空資料夾初始化新專案

這次把 #07 套用到新專案 `ours9638-create/codex-data`，實測流程如下：

- 工作資料夾一開始只有 Firebase 設定檔：`.firebaserc`、`firebase.json`、`firestore.rules`，不是 Git repo。
- 先補三個全域 skills：`startup-sync`、`shutdown-sync`、`project-init-sync`；建立後要重啟 Codex Desktop 才會被正式載入。
- 建立或補齊專案檔：`AGENTS.md`、`README.md`、`.gitignore`。
- Obsidian 駕駛艙建立在 vault 內：`codex-data/專案工作流程.md`，不要建在工作資料夾。
- 初始化 Git 後先 commit，再把 branch 改成 `main`。
- 建立 GitHub repo 並 push；若要 GitHub Pages，repo 需要公開，私有 repo 可能因帳號方案無法啟用 Pages。
- GitHub Pages API 建立時要使用巢狀欄位：
  ```powershell
  gh api --method POST repos/<owner>/<repo>/pages -f 'source[branch]=main' -f 'source[path]=/'
  ```
  不要寫成 `-f source.branch=main -f source.path=/`，那會回 `Invalid input`。

本次踩坑：

- `quick_validate.py` 需要 `PyYAML`。如果系統 Python 缺套件，會出現 `ModuleNotFoundError: No module named 'yaml'`；可先執行 `python -m pip install --user PyYAML`。
- Windows Python 預設編碼可能是 `cp950`，驗證含中文的 `SKILL.md` 會出現 `UnicodeDecodeError`；執行前加 `$env:PYTHONUTF8='1'`。
- 某些 PowerShell / New-Item 版本不支援 `-LiteralPath`；建立資料夾可改用 `New-Item -ItemType Directory -Force -Path ...`。
- Google Drive 同步資料夾內執行 `git branch -M main` 可能卡住，留下 `.git/packed-refs.lock`。先確認沒有 Git 程序，再清掉 stale lock 後重試。
- 若真的有殘留 `git.exe` 程序，不要直接刪 lock；先等候或確認是卡住程序，再處理。
- 私有 repo 啟用 GitHub Pages 可能回 `Your current plan does not support GitHub Pages for this repository.`。要嘛維持私有並不用 Pages，要嘛改公開 repo。
- 專案用途如果使用者只給很短的字串，例如 `codex_`，要提醒這會讓專案邊界不清楚；可先照填，但建議補成完整用途。
- 若 Codex 對原懶人包 repo 只有 `READ` 權限，不能直接 push；應 fork、開分支、push 到 fork，再開 PR 回原 repo。

## 常見問題

| 問題 | 原因 | 解法 |
|---|---|---|
| `gh repo create` 失敗 | GitHub CLI 未登入或權限不足 | 先跑 `gh auth status`，未登入就 `gh auth login` |
| GDrive 內 commit 報錯 | Google Drive 同步與 Git ref 寫入衝突 | 跑 `git config windows.appendAtomically false` 後重試 |
| 新 repo 跑 `git config windows.appendAtomically false` 出現 `fatal: not in a git directory` | 還沒初始化 Git repo | 先 `git init`，再跑 `git config windows.appendAtomically false` |
| 文件寫 main 但 Git 顯示 master | Git 初始 branch 尚未改名 | 第一次 commit 後執行 `git branch -M main` |
| `git branch -M main` 出現 `.git/packed-refs.lock` | Git 程序卡住或 Google Drive 留下 stale lock | 先確認沒有 Git 程序；必要時清掉 stale lock 後重試 |
| 「開工」沒觸發 | `startup-sync` skill 不存在或 Codex 尚未重啟 | 確認 `C:\Users\<你>\.codex\skills\startup-sync\SKILL.md` 存在，然後重啟 Codex |
| 「收工」沒觸發 | `shutdown-sync` skill 不存在或 Codex 尚未重啟 | 確認 `C:\Users\<你>\.codex\skills\shutdown-sync\SKILL.md` 存在，然後重啟 Codex |
| 「新專案初始化」沒觸發 | `project-init-sync` skill 不存在或 Codex 尚未重啟 | 確認 `C:\Users\<你>\.codex\skills\project-init-sync\SKILL.md` 存在，然後重啟 Codex |
| `quick_validate.py` 回 `ModuleNotFoundError: No module named 'yaml'` | Python 缺 `PyYAML` | 執行 `python -m pip install --user PyYAML` 後重試 |
| `quick_validate.py` 讀中文 `SKILL.md` 回 `UnicodeDecodeError` | Windows Python 預設用 `cp950` 讀 UTF-8 檔 | 執行前加 `$env:PYTHONUTF8='1'`，再跑 validator |
| `New-Item -LiteralPath` 不可用 | PowerShell 版本或命令相容性問題 | 改用 `New-Item -ItemType Directory -Force -Path ...` |
| Obsidian 讀不到 | MCP 設定未載入或 vault 路徑錯 | 確認 `~/.codex/config.toml` 有 `[mcp_servers.obsidian]`，改完要重開 Codex |
| 專案駕駛艙被建在工作資料夾 | 把 Obsidian 駕駛艙誤當成 repo 內文件 | 改成建在 Obsidian vault，例如 `Secondbrain/<專案名稱>/專案工作流程.md`，repo 內 AGENTS.md 只記 vault 與相對路徑 |
| 只寫 `my-project/專案工作流程.md`，不知道是哪個 vault | 缺少 vault 絕對路徑 | AGENTS.md 要同時記 `Obsidian vault` 絕對路徑與 `專案駕駛艙` vault-relative path |
| Firebase 工具看不到 | Firebase MCP 未設定或 Codex 尚未重啟 | 先跑 #04.5；Windows 建議 MCP command 用 `npx.cmd`，改完要重開 Codex |
| 私有 repo 開 GitHub Pages 失敗 | 目前 GitHub 方案可能不支援私有 repo Pages | 維持私有就不用 Pages；若一定要 Pages，改公開 repo 或調整方案 |
| GitHub Pages API 回 `Invalid input` | `source` 欄位格式錯 | 用 `-f 'source[branch]=main' -f 'source[path]=/'` |
| 不想公開 repo | 不需要 GitHub Pages 或不想公開內容 | 建立 private repo；若未來要 Pages，再評估是否公開 |
| 要跨電腦 | 工作檔與設定同步分屬不同系統 | 工作檔靠 GDrive / OneDrive，同步設定靠 chezmoi；新電腦先跑 Codex 環境懶人包 |
| 下載的 `.md` 打開是亂碼 | 下載檔或讀取工具的編碼不一致 | 回 GitHub raw 原始檔讀取，不要直接依亂碼版本修改 |
| 已有專案要套 #07 | 不是空資料夾，已有規則與歷史 | 先檢查現況，只補缺少的 `README.md`、`.gitignore`、Obsidian 紀錄等，不要覆蓋既有設定 |
| `AGENTS.md` 顯示未追蹤 | 舊專案可能原本只有 `CLAUDE.md`，或 AGENTS 尚未納入 Git | 若 #07 要讓 Codex 使用 AGENTS，確認內容無敏感資訊後納入本次 commit |
| `.gitignore` 註解或中文顯示亂碼 | 舊檔案編碼或終端顯示問題 | 可保留規則本體，將短檔案整理成 UTF-8、繁中註解清楚的安全版本 |
| README 缺少 | 舊示範專案可能直接靠 HTML 與 AGENTS 運作 | 補一份簡短 README：用途、主要檔案、工作模式、安全原則即可 |
| 專案用途太短，例如只寫 `codex_` | Codex 難以判斷專案邊界 | 可先照填，但要提醒使用者之後補成一句完整用途 |
| GitHub API 更新中文檔名失敗 | PowerShell / Node 管線可能把中文檔名轉成問號 | 對 GitHub contents API 使用 URL encoded path，或用純 PowerShell Unicode 流程 |
| GitHub 文件被壓成一長行 | 讀 raw 時沒有保留換行 | 用 `$Lines = gh api ...` 後 `$Content = $Lines -join "`n"` 再更新 |
| 對原懶人包 repo 只有 `READ` 權限 | 不是 repo collaborator 或 owner | fork repo、開分支、push 到 fork，再開 PR 回原 repo |

## 安全與隱私

- `.codex/`、`.claude/` 不要 commit
- `.env`、API key、token、密碼不要 commit
- Firebase 前端 config 可以公開，但 Admin 憑證不可以
- 學生資料只存座號與班級代號，不存真名
- AGENTS.md 記規則；Obsidian 記進度；不要雙寫漂移
