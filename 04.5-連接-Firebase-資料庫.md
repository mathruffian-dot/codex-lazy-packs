> - 建立

---

### 步驟二之二：設定 Firestore 安全規則（自動化）

請 Codex 在使用者的專案資料夾建立三個檔案：

**1. `firestore.rules`**：
```
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    match /wordcloud_words/{document} {
      // 測試用公開白名單：只適合課堂 demo，不可直接用在正式學生資料或長期公開服務。
      allow read, write: if true;
    }
    match /{document=**} {
      allow read, write: if false;
    }
  }
}
```

**2. `firebase.json`**：
```json
{
  "firestore": {
    "rules": "firestore.rules"
  }
}
```

**3. `.firebaserc`**：
```json
{
  "projects": {
    "default": "[使用者的 Firebase 專案 ID]"
  }
}
```

部署規則：
```powershell
npx.cmd -y firebase-tools@latest deploy --only firestore:rules
```

成功會看到 `Deploy complete!`。

> 💡 之後新工具：測試階段可在 `firestore.rules` 加 `match /votes/{document} { allow read, write: if true; }` → 對 Codex 說「部署 Firestore 規則」。正式服務請改成登入、班級碼或教師管理條件，不要沿用公開寫入規則。

## 階段二：連接 MCP

### 步驟三：把 Firebase 註冊為 Codex 的 MCP server

> ✅ 三種 Codex 共用 `~/.codex/config.toml`。

**方法 A：Codex Desktop GUI（推薦）**

1. Desktop → 設定 → **Integrations & MCP** → **Add server**
2. 填：
   - Name：`firebase`
   - Command：Windows 建議填 `npx.cmd`；macOS / Linux 填 `npx`
   - Args：`-y`、`firebase-tools@latest`、`mcp`
3. 儲存

**方法 B：手動編輯 `~/.codex/config.toml`**

```toml
[mcp_servers.firebase]
command = "npx.cmd"
args = ["-y", "firebase-tools@latest", "mcp"]
startup_timeout_sec = 60
tool_timeout_sec = 120
```

> Windows 建議用 `npx.cmd`，比較不會踩到 PowerShell `.ps1` 執行原則問題。macOS / Linux 可用 `command = "npx"`。

**方法 C：CLI**

```bash
codex mcp add firebase -- npx -y firebase-tools@latest mcp
```

> 2026-04-29 實測：本機 Codex Desktop 可用「方法 B」成功載入。若 `codex` CLI 在 Windows 顯示 `Access is denied`，不要卡在 CLI，改用 GUI 或手動改 `config.toml`。

### 步驟四：登入 Firebase CLI

```powershell
npx.cmd -y firebase-tools@latest login
```

> 🖐️ 瀏覽器會開 Google 登入頁，授權即可。

驗證：
```powershell
npx.cmd -y firebase-tools@latest projects:list
```

> ⚠️ `firebase login` 必須在**互動式終端**執行。如果在 Codex 對話中卡住，請使用者打開 cmd / PowerShell 手動跑一次。

> ⚠️ `projects:list` 列出的才是目前登入帳號真正有權限管理的專案。若專案資料夾內的 `.firebaserc`、HTML `firebaseConfig` 指向另一個專案，不要直接部署；先確認是否登入錯 Google 帳號，或決定統一改用目前可管理的專案。

測試情境：

- 同步更新 `.firebaserc` 與所有 HTML 的 `firebaseConfig`，避免規則部署到 A 專案、網頁卻連 B 專案。
- 保留既有多集合白名單規則，成功部署到新專案；未列入白名單的集合仍會回 `403`。
- Firebase MCP 成功完成完整 CRUD：列出集合、新增、讀取、更新、再次讀取、刪除，最後確認文件不存在。
- 測試資料使用專用文件 ID，驗證完成後立即刪除，沒有碰其他資料。

本次踩坑：
- **舊設定不等於目前可管理的專案**：`.firebaserc`、`AGENTS.md` 或 HTML 可能保留舊帳號的 project ID。應以 `projects:list` 的實際權限為起點，再決定登入正確帳號或統一遷移。
- **不要只改 `.firebaserc`**：它只控制 CLI 部署目標；前端真正連去哪裡由 HTML 的 `firebaseConfig` 決定。兩邊不同步會形成「規則部署成功，但網頁仍操作舊資料庫」。
- **不要直接套用懶人包的單集合範例覆蓋既有規則**：既有專案可能已有多個教學工具集合。應保留原本資料結構，只校正專案目標與必要規則。
- **MCP 加入後要完整重啟 Codex**：同一輪對話通常不會自動載入新工具。可先用 Firebase CLI 完成登入與部署，重啟後再用 MCP 做 CRUD 驗證。
- **空資料庫回 `{}` 是正常狀態**：先新增測試文件，再列集合與文件；不能只因空結果就判定連線失敗。
- **部署成功不等於資料庫可操控**：最可靠的驗證是完整 CRUD，並在最後確認測試文件已刪除。
- **PowerShell 組 REST URL 時要用 `${變數}` 明確界定名稱**：例如 `"${Base}?documentId=${DocId}"`。直接寫 `"$Base?documentId=..."` 可能被錯誤解析，測試腳本還應設定 `$ErrorActionPreference = 'Stop'`，避免前一步失敗後仍輸出誤導性的成功訊息。

## 如果失敗

對 Codex 說：「Firebase 懶人包失敗，幫我檢查重新處理。」

完全重置：
- **Desktop**：設定 → Integrations & MCP → 刪 `firebase`
- **手動**：編輯 `~/.codex/config.toml` 刪 `[mcp_servers.firebase]` 段
- **CLI**：`codex mcp remove firebase`

重登 Firebase：
```powershell
npx.cmd -y firebase-tools@latest logout
npx.cmd -y firebase-tools@latest login
```

## 常見問題

| 問題 | 解法 |
|------|------|
| `npx: command not found` | 確認 Node.js 已裝，重啟終端機 |
| PowerShell 顯示 `npx.ps1 cannot be loaded because running scripts is disabled` | 改用 `npx.cmd --version`、`npx.cmd -y firebase-tools@latest ...` |
| Codex 讀 Google Drive 專案資料夾出現 sandbox 權限錯誤 | 同意授權後重跑；不要先判定檔案壞掉 |
| 查詢失敗 | 確認 `firebase login` 已成功 |
| `projects:list` 看不到專案資料夾原本指定的 project ID | 可能登入了不同 Google 帳號，或舊專案不屬於目前帳號。先確認帳號；若決定改用可管理的新專案，要同步更新 `.firebaserc`、HTML `firebaseConfig` 與專案文件 |
| `.firebaserc` 已改，但網頁仍寫入舊資料庫 | `.firebaserc` 只影響 CLI 部署；網頁連線目標由 `firebaseConfig` 決定，兩者必須同步 |
| 列集合或文件只回 `{}` | 資料庫目前是空的。新增一筆測試文件後再列出；集合會在第一筆文件建立後出現 |
| `firestore_query_collection` 回 `read_time cannot be in the future` | 可能是 MCP 查詢工具的時間參數問題。先用 `firestore_get_document` 或 `firestore_list_documents` 驗證可讀取，必要時重啟 MCP 後再查 |
| 看不到 Firestore 資料 | 確認 Console 已啟用 Firestore |
| 安全性規則過期 | 用了測試模式。改白名單規則（步驟二之二） |
| `Permission denied` | rules 沒允許這個集合 |
| `Collection id is invalid because it contains /` | `collection_path` 別含尾巴 `/`，寫 `wordcloud_words` 不要 `wordcloud_words/` |
| `firebase login` 在對話中卡住 | 開 cmd/PowerShell 手動跑 `npx firebase-tools login`（互動式登入無法在 Codex 對話內完成） |
| Codex 重開前看不到 Firebase 工具 | 改完 `~/.codex/config.toml` 後，要完整重開 Codex Desktop；只在同一輪對話重新搜尋通常不會載入新 MCP server |
| Windows 跑 `npx` 出問題 | 把 MCP 設定的 `command` 改成 `npx.cmd` |
| `codex --version` 顯示 `Access is denied` | 不影響 Codex Desktop 實測。改走 GUI 或手動編輯 `~/.codex/config.toml` |
| `firestore_add_document` 說 document 格式 invalid | 不要把 Firestore document 包成 JSON 字串；要傳 document 物件格式，例如 `fields.message.stringValue` |
| 規則顯示 `Deploy complete!`，但不知道是否真的可用 | 再做一次新增、讀取、更新、刪除與刪除後讀取的完整 CRUD 測試 |
